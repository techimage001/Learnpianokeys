# Learn Piano Keys - V3

Static site plus a small PHP lead collector. No build dependencies, no
frameworks, no paid APIs, no sampled audio.

## Build

    node tools/build.js      # regenerates every .html from src/ + sitemap + manifest
    node tools/tests.js      # static assertions across every page family
    node tools/smoke.js      # BOOTS every page in a DOM and clicks every control

Run all three before every release. `tests.js` reads the files; `smoke.js`
executes them.

**Why smoke.js exists.** `node --check` validates syntax only. A refactor once
deleted the practice room's entire state object and every helper function in
one slice, the syntax check passed, the static tests passed, and the page was
completely dead in a real browser for two releases. Only running the code
catches that. `smoke.js` loads each page in jsdom, evaluates every script in
one shared scope exactly as a browser does, asserts the app actually built
itself, and clicks every button on every page.

Never hand-edit the .html files at the repo root. They are generated.
Edit `src/*.html` for body content and `tools/build.js` for head, nav and footer.

## Releasing

1. Bump `ASSET_V` in `tools/build.js`.
2. `node tools/build.js && node tools/tests.js`
3. Commit and push. Hostinger pulls from GitHub as usual.
4. Hard refresh once, then check one page from each family.

The test harness fails the build if two pages ever end up on different asset
versions, which is the bug that silently shipped stale CSS on earlier projects.

## The reading feature

`/how-to-read-music.html` is the reading hub and the note trainer. It targets
the "how to read music" cluster, which is a bigger head term than any single
piece of repertoire.

It is written to be extractable by AI overviews as well as ranked: every
section opens with a `.answer-first` paragraph that answers the section's
question completely on its own, in plain sentences, without needing the
surrounding page. The generator emits a `HowTo` graph with six steps alongside
the usual `FAQPage`, and the harness fails the build if a step loses its URL or
its answer text.

The stave renderer in `assets/reading.js` is deliberately self-contained rather
than shared with the practice room, so adding this feature could not regress
`app.js`.

## Getting indexed, everywhere and not just Google

### 1. Verify ownership with each engine

Open `tools/build.js` and fill in the `VERIFY` object with the value each
webmaster tool gives you, then rebuild. Empty entries emit no tag at all, so
there are never dangling meta tags on the page.

    google-site-verification   Google Search Console       search.google.com/search-console
    msvalidate.01              Bing Webmaster Tools        bing.com/webmasters
    yandex-verification        Yandex Webmaster            webmaster.yandex.com
    naver-site-verification    Naver Search Advisor        searchadvisor.naver.com
    baidu-site-verification    Baidu Ziyuan                ziyuan.baidu.com
    seznam-wmt                 Seznam Webmaster            search.seznam.cz/wmt
    p:domain_verify            Pinterest                   pinterest.com/settings/claim

Bing Webmaster Tools can import everything straight from Search Console once
Google is verified, which is by far the fastest route. Do Google first.

Submit `https://learnpianokeys.com/sitemap.xml` in each tool. DuckDuckGo,
Ecosia, Yahoo and most other engines take their index from Bing, so verifying
Bing covers them without any extra work.

### 2. IndexNow, for everything except Google

One submission notifies Bing, Yandex, Seznam, Naver and Yep at once.

    node tools/submit-index.js                      every URL in the sitemap
    node tools/submit-index.js /tools.html          just the pages that changed

The key lives at `79e2624f7b502947f1147e5f7b9c8dd2.txt` in the site root and is
regenerated by the build. **Do not change the key once the site is live**: the
file and the submission must match or every engine returns 403.

Run the submit script after each deploy, not before, or the engines will crawl
the old version.

### 3. Google

Google does not take part in IndexNow, and the old sitemap ping endpoints at
`google.com/ping` and `bing.com/ping` were both retired, so any script claiming
to use them is doing nothing. For Google the sitemap does the work. For anything
urgent, use URL Inspection then Request Indexing on that one page.

Expect Search Console to say "Processing data, please check again in a day or
so" for several days while pages are already indexed and visible via a `site:`
search. The index is live; the reports lag. Cached titles also persist until
Google recrawls.

## Server setup (once)

The lead collector needs a private folder OUTSIDE public_html:

    domains/learnpianokeys.com/lpk_private/

Copy `api/secrets.example.php` to `lpk_private/secrets.php` and fill in:

    admin_password   long random string, this is the login for /api/leads.php
    salt             long random string, used to hash IPs
    smtp_host        smtp.hostinger.com
    smtp_port        587
    smtp_user        info@learnpianokeys.com
    smtp_pass        that mailbox's password

`lpk_private/` must be writable by PHP so `leads.sqlite` can be created there.

If secrets.php is missing the site still works and the admin panel simply
refuses to open. It fails safe.

### Email deliverability is a hosting job, not a code job

Verification links land in spam unless all of this is done in hPanel and DNS:

- create the `info@learnpianokeys.com` mailbox and use its password above
- SPF record for Hostinger's mail servers
- DKIM enabled for the domain
- a DMARC record, `p=none` to begin with

Without authenticated SMTP the code falls back to PHP `mail()`, which mostly
gets filtered. Do not skip this step.

## Signup, and what is honestly free

Nowhere on the site claims there is no signup, because there is one. What is
true, and what the copy says:

- The beginner walkthrough, all four tools and the practice tracker ask
  nothing of anyone, ever.
- The practice room asks for an email after three sessions.
- Signing up is free, nothing is charged and no card details are requested.
- The first time the card appears it can be dismissed. After that it stays.
- Progress is stored in the browser whether or not you sign up.

`tools/tests.js` fails the build if any page ever says "no account", "no
signup" or "no email" again.

## The gate

Three practice-room sessions, then a signup card. The beginner walkthrough,
all four tools, and the practice tracker are never gated, because those are
the pages search traffic lands on. An address already on the list unlocks a
new device immediately.

## Things worth remembering

- Web MIDI needs HTTPS. localhost is exempt.
- The microphone is monophonic by design and is labelled approximate in the UI.
- All progress lives in `localStorage` under `lpk.store.v1`. No account.
- Every piece is public domain and all audio is synthesised. Nothing sampled.
- No competitor is named anywhere on the site.
- There is no advertising anywhere on this site and no ad network code of any
  kind. No `ads.txt`, no ad tags, no ad slots. The privacy page says so plainly
  and the test harness fails the build if an ad script or `ads.txt` reappears.
- 404s now return a real 404 status from `/404.html`. They used to serve the
  homepage with a 200, which Google counts as a soft 404 and which would have
  masked a missing sitemap.
- Brass buttons inside the nav need their own colour rule. `.nav a` outranks
  `.btn-primary` on specificity, which is what made the header button
  unreadable. There is a test asserting it now.
